# Business OS 技术架构设计

> 本文档记录了 Business OS 从产品设计到技术实现的完整架构讨论与决策。

---

## 一、产品定位

**一句话概括**：让中小企业老板通过对话把脑子里的经营方法论（选品、定价、客服考核等）固化成可执行的 Skill，由 AI 按企业自定义方法论自动完成工作流。

### 核心演进路径

```
Phase 1: 对话沉淀企业工作流为 Skill
Phase 2: Skill 按计划/事件自动执行，日常任务自转
Phase 3: 平台持有企业运营全貌 → 企业数字孪生
```

系统不只是一个 "AI 助手"，而是一个**持续运转的企业经营自动机**。Skill 不只是 "被调用的函数"，而是 "持续运行的守护进程"。

## 三、技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **前端** | React 18 + Next.js 14 + TailwindCSS + TypeScript + Zustand | 流式渲染好、SSR支持、生态成熟、状态管理轻量 |
| **后端网关** | Java 17 + Spring Boot 3.2 (Web, JPA, Security, WebFlux) | 企业级稳定、生态成熟、JPA简化数据层、WebFlux支持响应式流 |
| **AI引擎** | Python 3.11 + FastAPI | AI生态最成熟、异步性能好、与LLM SDK兼容 |
| **大模型** | 通义千问 (Qwen) via DashScope | 中文能力强、成本可控、保留切换其他模型的能力 |
| **结构化存储** | PostgreSQL 16 | Skill元数据、用户、企业资产、执行记录、数字孪生状态 |
| **缓存/会话** | Redis 7 | 对话上下文、执行状态、Session |
| **通信协议** | NDJSON (Python↔Java), SSE (Java↔前端), JWT鉴权 | NDJSON 解析可靠、SSE 流式自然、JWT 无状态鉴权 |

### 关于网关层的决策

**结论：能力必须有，但不需要独立的网关服务，用分层中间件实现。**

| 能力 | 是否必须 | 实现方式 | 原因 |
|------|---------|----------|------|
| **鉴权** | 必须 | JWT + Spring Security Filter | 企业数据涉及商业机密 |
| **多租户隔离** | 必须 | TenantContext (请求级上下文) + ORM层强制过滤 | 产品根基，数据绝不能串 |
| **会话管理** | 必须 | Redis + PostgreSQL 双层 | 对话上下文 + Skill执行状态 |
| **限流** | 需要但不紧急 | 预留接口，后续按企业配额限流 | 初期用户少，但LLM调用成本高 |

多租户隔离设计要点：

- 所有数据表都带 `enterprise_id` 字段
- ORM 层通过 `TenantContext` 自动注入租户过滤条件
- 任何查询都不可能意外访问其他租户的数据
- 文件存储按 `/{enterprise_id}/` 目录隔离

## 四、核心模块设计

### 1. Skill 体系

#### Skill 的四件套

- **skill.yaml** — 方法论（怎么做，分几步，每步的 prompt/tools/checkpoint）
- **reference/** — 参考文档（每步的细化规则）
- **tools** — 执行脚本（取数、计算、生成的Tool函数）
- **assets/** — 企业资产（私有数据、偏好、约束）

#### Skill 的三种运行模式

```
① 对话触发（on-demand）
   用户说"帮我看看这个款" → 执行一次

② 定时执行（scheduled）
   询盘日报 → 每天8:00自动跑
   巡盘周报 → 每周一9:00自动跑

③ 事件驱动（event-triggered）[预留]
   新询盘进来 → 自动识别大客户 → 推送通知
   竞品上新 → 自动拉取分析 → 推送预警
```

#### Skill 的生命周期

```
┌─────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│ 沉淀 │ →  │ 确认 │ →  │ 试跑 │ →  │ 上线 │ →  │ 自转 │
└─────┘    └──────┘    └──────┘    └──────┘    └──┬───┘
  对话/上传    用户确认     手动触发     设定自动化    │
  提取方法论   Skill结构    验证结果     调度规则     │
                                                   │
    ┌──────────────────────────────────────────────┘
    │
    ▼
┌──────┐    ┌──────┐    ┌──────┐
│ 监控 │ ←→ │ 迭代 │ ←→ │ 沉淀 │
└──────┘    └──────┘    └──────┘
  执行结果      用户反馈      更新数字孪生
  异常告警      自动优化      积累决策记忆
```

#### Skill 执行状态机（SkillRunner）

```
START → INTAKE (信息采集)
      → CONFIRM_PLAN (确认执行计划)
      → EXECUTING (分步执行，含checkpoint)
      → CAPTURE (沉淀偏好/记忆)
      → DONE
```


### 通信协议：NDJSON 事件类型

Python → Java 之间使用 NDJSON（Newline Delimited JSON）流式通信：

| 事件类型 | 说明 |
|----------|------|
| `token` | LLM 流式 token |
| `intake_questions` | Skill 信息采集问题 |
| `plan_preview` | 执行计划预览 |
| `step_start` / `step_done` | 步骤开始/完成 |
| `checkpoint` | 需用户确认的检查点 |
| `capture_offer` | 偏好沉淀提议 |
| `tool_call` | Tool 调用事件 |
| `skill_done` | Skill 执行完成（含 execution_record） |
| `memory_save` | 偏好保存事件 |
| `skill_created` | 新 Skill 创建完成 |
| `done` | 流结束 |

```
       │

       ├── 保存 preferences 为 EnterpriseAsset
       └── 发送 SSE 通知
              │
              ▼
```

## 八、关键技术决策记录

| 决策点 | 选择 | 原因 |
|--------|------|------|
| Lombok vs 手写 | 手写 getter/setter/builder | JDK 25 兼容性问题，手写更稳定 |
| 流式协议 | NDJSON (非 SSE) | Python→Java 解析更可靠，支持结构化事件 |
| 前端→后端流式 | SSE | 浏览器原生支持，单向推送足够 |
| 多租户方案 | 共享DB + enterprise_id + ORM层强制过滤 | 初期简单有效，后续可升级为Schema隔离 |
| Agent框架 | 自研（参考LangGraph思路） | 渐进式上下文、动态Tool注册等需求框架难满足 |
| Skill存储 | YAML定义 + PostgreSQL元数据 | 结构清晰、人可读、易版本控制 |
| 网关策略 | 中间件链（不独立部署） | 初期团队小、用户少，避免过度工程化 |
| 大模型 | 通义千问(主)，保留切换能力 | 中文能力强、成本可控、工厂方法模式易切换 |

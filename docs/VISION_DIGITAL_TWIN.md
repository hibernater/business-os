# 从 AI 助手到企业数字孪生：核心设计思路

> 本文档记录了 Business OS 产品愿景的一次关键转变——从"响应式 AI 工具"到"企业经营自动机"再到"企业数字孪生"的三阶段演进。这一思路贯穿了整个技术架构的设计。

---

## 一、讨论背景

在初始技术架构讨论中，系统被设计为一个"用户问一句，系统跑一次"的**响应式 AI 助手**。但产品创始人指出了一个根本不同的方向：

> "这是一个试图把中小企业的日常工作流自动执行起来的 AI 原生平台，用户是产业带中小企业，工作流用 Skill 来承接，产品先通过对话等交互把企业的工作流沉淀下来，然后自动执行，进而变成企业的数字孪生。所以重点是企业的日常任务自动执行起来，一是帮助企业 AI 提效，二是企业大部分日常任务自动执行之后，这个产品会演变成企业的数字孪生。"

这段话改变了架构的核心驱动力。

## 二、核心思路转变

```
之前理解：用户提问 → 系统执行 Skill → 返回结果（响应式）

现在理解：
  Phase 1: 对话沉淀企业工作流为 Skill
  Phase 2: Skill 按计划/事件自动执行，日常任务自转
  Phase 3: 平台持有企业运营全貌 → 企业数字孪生
```

**本质区别**：系统不只是一个"AI 助手"，而是一个**持续运转的企业经营自动机**。Skill 不只是"被调用的函数"，而是"持续运行的守护进程"。

### 两层价值

1. **AI 提效**（Phase 1-2）：企业的日常重复性任务（询盘日报、经营复盘、客服考核、退款归因……）由 AI 自动完成，老板从执行者变成审阅者。

2. **企业数字孪生**（Phase 3）：当大部分日常 Skill 都在自动运转，并且执行结果持续回写企业状态模型时，系统逐步拥有了企业经营的完整镜像——它知道企业卖什么、卖给谁、怎么卖、效果如何。此时系统不再是工具，而是**比老板更全面了解企业状况的数字孪生体**。

## 四、Skill 的重新定义

### 在这个思路下，Skill 不是"可调用的工具"，而是"企业经营的数字化代理"

每个 Skill 持续运转，它的输入是业务事件和企业数据，它的输出不仅是交付物，更是企业数字孪生的一次更新。当所有核心 Skill 都在自动运转，并且相互之间形成依赖网络时，这个系统就不再是一个"AI助手"，而是一个真正的"企业数字孪生"。

### Skill 的三种运行模式

```
┌─────────────────────────────────────────────────┐
│           Skill 的三种运行模式                     │
│                                                  │
│  ① 对话触发（on-demand）                         │
│     用户说"帮我看看这个款" → 执行一次             │
│                                                  │
│  ② 定时执行（scheduled）                         │
│     询盘日报 → 每天8:00自动跑                     │
│     巡盘周报 → 每周一9:00自动跑                   │
│     客服考核 → 每月1号自动跑                      │
│                                                  │
│  ③ 事件驱动（event-triggered）                   │
│     新询盘进来 → 自动识别大客户 → 推送通知         │
│     竞品上新 → 自动拉取分析 → 推送预警             │
│     退货发生 → 自动归因分类 → 写入月报             │
│     Skill A 完成 → 触发 Skill B 执行              │
└─────────────────────────────────────────────────┘
```

### Skill 的生命周期

不再是"创建→调用→结束"，而是一个持续演进的闭环：

```
┌─────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│ 沉淀 │ →  │ 确认 │ →  │ 试跑 │ →  │ 上线 │ →  │ 自转 │
└─────┘    └──────┘    └──────┘    └──────┘    └──┬───┘
  对话/上传    用户确认     手动触发     设定自动化    │
  提取方法论   Skill结构    验证结果     调度规则     │
                                                   │
    ┌──────────────────────────────────────────────┘
    │
    ▼
┌──────┐    ┌──────┐    ┌──────┐
│ 监控 │ ←→ │ 迭代 │ ←→ │ 沉淀 │
└──────┘    └──────┘    └──────┘
  执行结果      用户反馈      更新数字孪生

```

### 这个架构与"响应式 AI 助手"的根本区别

| 维度 | 响应式 AI 助手 | 企业经营自动机（本设计） |

| Skill 本质 | 被调用的函数 | 持续运行的守护进程 |
| 执行结果去向 | 返回给用户就结束 | 返回用户 + 写入Memory Bus + 更新状态模型 |
| Skill 关系 | 彼此独立 | DAG 依赖图，上游触发下游 |

| 数据库设计 | Skill + Asset | + enterprise_state + decision_records + skill_events |
| 用户角色 | 提问者 | 审阅者（大部分任务自动完成，用户只看结果和异常） |
| 终极形态 | 智能客服 | 企业数字孪生 |

## 六、核心模块设计

### 1. Automation Scheduler（自动调度中心）

让企业日常任务"自转"的核心。

```python
class SkillScheduleConfig:
    skill_id: str
    mode: Literal["on_demand", "scheduled", "event_triggered"]

    # 定时模式
    cron_expr: str | None          # "0 8 * * *" 每天8点

    # 事件驱动模式
    trigger_events: list[str]      # ["new_inquiry", "competitor_new_product"]
    trigger_conditions: dict       # {"buyer_level": ">=A", "amount": ">5000"}

    # 通用
    auto_execute: bool             # True=静默执行，False=推送确认后执行
    notify_on_complete: list[str]  # 完成后通知渠道 ["wechat", "dingtalk"]
    retry_policy: RetryPolicy      # 失败重试策略
    depends_on: list[str]          # 依赖的前置Skill（DAG编排）

class AutomationScheduler:
    """企业日常任务的自动调度中心"""

    async def register(self, config: SkillScheduleConfig):
        """注册一个自动化任务"""
        match config.mode:
            case "scheduled":
                self.cron_engine.add_job(config)
            case "event_triggered":
                self.event_bus.subscribe(config.trigger_events, config)

    async def on_event(self, event: BusinessEvent):
        """业务事件发生时，检查是否有Skill需要触发"""
        matched = self.event_bus.match(event)
        for config in matched:
            if config.auto_execute:
                await self.execute(config, event.payload)
            else:
                await self.notify_user_for_confirm(config, event)

    async def execute(self, config: SkillScheduleConfig, input_data: dict):
        """执行Skill并回写结果到数字孪生"""
        result = await self.skill_runner.execute(config.skill_id, input_data)
        # 关键：每次执行的结果都写入企业状态模型
        await self.digital_twin.update(config.enterprise_id, result)
        await self.notify(config, result)
```

### 2. Enterprise Digital Twin（企业数字孪生层）

整个产品的终极价值承载。不是一个独立功能，而是所有 Skill 执行结果的**汇聚与结构化**。

class DecisionRecord:
    """一次经营决策的完整记录 —— 数字孪生的'记忆'"""

    recommendation: dict   # Skill给出的建议
    user_choice: dict      # 用户实际选择

### 3. Enterprise Memory Bus（企业记忆总线）

让数字孪生"活起来"的关键管道。每次 Skill 执行都不只是完成任务，而是同时在更新企业的数字孪生。

## 八、关键技术决策调整

这个思路导致了以下架构层面的关键变化：

| 决策点 | 之前方案（响应式） | 调整后方案（自动机） | 原因 |
|--------|------------------|-------------------|------|
| Skill执行结果处理 | 返回给用户就结束 | 返回用户 + 写入Memory Bus + 更新状态模型 | 每次执行都在构建数字孪生 |
| Skill运行模式 | 只有on-demand | on-demand + scheduled + event-driven | 日常任务自转的基础 |
| 数据库设计 | Skill+Asset为主 | 增加 enterprise_state + decision_records + skill_events | 数字孪生的数据底座 |
| 前端架构 | 纯对话UI | 对话 + 驾驶舱(Dashboard) | 从"交互工具"变成"经营系统" |
| 消息队列 | 可选 | 必须 | 事件驱动架构的核心 |
| Skill之间的关系 | 独立 | DAG依赖图 | 上游Skill完成可自动触发下游 |

## 九、需要为后续埋的桩

即使初期只做手动触发，架构上必须预留：

1. **Skill执行结果的结构化回写接口** — 每次执行不只是返回文本，而是同时产出 `state_updates` 和 `decision_record`。初期可以只存不用，但数据结构要定好。

2. **事件总线的基座** — 哪怕初期只用来做简单的内部通信，也要把 Event Bus 的接口定义好，后续直接在上面加调度逻辑。

3. **Skill 的 `schedule_config` 字段** — 数据模型里预留，初期默认 `manual`，后续开放配置。

4. **enterprise_state 表** — 五维状态模型的数据库表初期即创建，Skill 执行时逐步回写。
